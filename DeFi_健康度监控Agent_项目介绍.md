# DeFi 协议健康度监控 Agent — 项目介绍

---

## 一、项目背景

### 1.1 赛道定位：科技金融 × 普惠金融

Decentralized Finance（DeFi）是区块链技术驱动的开放式金融体系，其核心理念是让任何人——无需银行账户、无需信用审核——通过互联网接入金融服务。截至 2026 年，DeFi 市场总锁仓价值（TVL）已超过千亿美元规模，涵盖借贷、交易、衍生品、稳定币等多种金融场景。

然而，去中心化带来的不仅是自由，还有**信息不对称和风险扩散**：

- **普通用户** 无法实时监控协议的健康状态（清算风险、流动性枯竭、预言机异常）
- **专业用户** 需要横跨多个平台（Etherscan、DefiLlama、Dune Analytics）手动收集数据
- **机构投资者** 缺少自动化的风险预警工具

本项目的目标是：**基于 Vibe Coding 方法论，利用 Claude Code（Anthropic 智能体 CLI 开发环境）+ DeepSeek V4 大模型，开发一个 DeFi 协议健康度监控 Agent**，以自然语言交互的方式，降低 DeFi 风险监控的门槛，同时覆盖科技金融（技术驱动的基础设施）和普惠金融（低门槛工具赋能普通用户）两个赛道。

### 1.2 核心痛点

| 痛点 | 描述 | 影响人群 |
|---|---|---|
| 信息碎片化 | 协议数据分布在多个平台和工具中，缺乏统一入口 | 所有 DeFi 用户 |
| 预警滞后 | 清算、脱锚、攻击等事件发生在数分钟内，人工无法 24h 监控 | 借贷用户、做市商 |
| 交互门槛高 | 链上数据查询需要懂 RPC、合约 ABI、Etherscan API | 普通散户、新用户 |
| 多链碎片化 | 同协议部署在多个公链，数据分散难以汇总 | 跨链用户、数据分析师 |

## 二、解决方案：智能体架构

### 2.1 什么是 Vibe Coding

Vibe Coding 是一种**以自然语言驱动开发**的新型编程范式。开发者用自然语言描述需求，AI（大语言模型）理解意图并自动生成代码、编排工具调用、组织数据流。本项目采用 **Claude Code（Anthropic 出品的 CLI 智能体开发环境）** 作为 Agent 运行与编排框架，调用 **DeepSeek V4** 作为核心推理引擎，实现"一句话问协议，Agent 自动查数据给答案"的交互体验。

> **选型理由**：Claude Code 原生支持 Function Call / Tool Use 机制，内置多轮对话上下文管理、工具链注册与调度，且与 DeepSeek V4 的 API 协议兼容。相比 TRAE 等较新的框架，Claude Code 社区生态更成熟，踩坑成本更低。DeepSeek V4 在中文金融语义理解上表现优异，且推理成本可控。

### 2.4 功能模块

#### 模块一：协议概览仪表盘
- 实时接入 DefiLlama API，展示 Top 20 协议 TVL 排名
- 单个协议深度数据：总锁仓价值、借贷总额、利用率、储备金率
- 多链数据汇总（Ethereum、Arbitrum、Optimism、Polygon 等）

#### 模块二：清算风险监控
- 自动计算各协议的清算健康度指标
- 设定预警阈值：利用率 > 80% 黄色预警，> 95% 红色预警
- 关联历史数据，展示健康度变化趋势

#### 模块三：自然语言分析 Agent（核心亮点）
- 支持开放式提问，示例：
  - *"Aave V3 的 TVL 和利用率是多少？"*
  - *"最近 24 小时有没有大额清算事件？"*
  - *"对比一下 Compound 和 Aave 的清算风险"*
- 支持追问和上下文关联

#### 模块四：异常事件检测
- TVL 突降检测（24 小时内降幅超过 10% 触发警报）
- 大额资金流动监控（单笔转账超过 100 万美元）
- 协议关键参数变更检测

## 三、技术实现方案

### 3.1 技术栈选型

| 层级 | 技术选型 | 说明 |
|---|---|---|
| AI Agent 开发环境 | Claude Code（Anthropic） | CLI 智能体编排、Tool Use 注册、多轮上下文管理 |
| 大语言模型 | DeepSeek V4（深度求索） | 自然语言理解、Function Call 工具调用、分析生成 |
| 后端服务 | Python FastAPI | API 服务、数据处理、工具函数实现 |
| 前端界面 | 微信小程序（WXML + ECharts） | 移动端数据可视化 + 自然语言交互入口 |
| 数据源 | DefiLlama API（免费，聚合 800+ 协议） | TVL、利用率、交易量等 |
| 部署方案 | 微信云托管 / Railway（备选） | 国内极速访问 / 海外兜底 |

### 3.2 数据接入

DefiLlama API 是本项目的核心数据源，提供免费且无需认证的接口：

| 接口 | 用途 | 频率限制 |
|---|---|---|
| `/protocols` | 获取所有协议列表和 TVL | 无公开限制 |
| `/protocol/{name}` | 单个协议详细数据 | 建议 1 次/秒 |
| `/charts/{name}` | TVL 历史走势 | 建议 1 次/10秒 |
| `/lends/{name}` | 借贷协议数据（利用率等） | 建议 1 次/10秒 |

### 3.3 Agent 工具注册（示例）

在 Claude Code 环境中，每个数据查询封装为一个 Python 工具函数，通过 Claude Code 的 Tool Use 机制注册并被 DeepSeek V4 调用：

```python
# tools/defi_api.py — 工具函数实现（FastAPI 后端）

import httpx
import asyncio
from typing import Optional

DEFILLAMA_BASE = "https://api.llama.fi"

async def get_protocol_tvl(name: str) -> dict:
    """获取指定 DeFi 协议的当前 TVL 及多链分布"""
    async with httpx.AsyncClient(timeout=10.0) as client:
        resp = await client.get(f"{DEFILLAMA_BASE}/protocol/{name}")
        resp.raise_for_status()
        data = resp.json()
        return {
            "name": data.get("name"),
            "tvl": data.get("tvl"),           # 单位: USD
            "chains": data.get("chains", []),  # 多链部署列表
            "url": data.get("url")
        }

async def get_lending_data(name: str) -> dict:
    """获取借贷协议的利用率、TVL、清算数据"""
    async with httpx.AsyncClient(timeout=10.0) as client:
        resp = await client.get(f"{DEFILLAMA_BASE}/lends/{name}")
        resp.raise_for_status()
        return resp.json()

async def get_tvl_history(name: str, days: int = 30) -> list:
    """获取协议 TVL 历史走势"""
    async with httpx.AsyncClient(timeout=15.0) as client:
        resp = await client.get(f"{DEFILLAMA_BASE}/charts/{name}")
        resp.raise_for_status()
        data = resp.json()
        # 取最近 N 天数据点
        return data[-days:] if len(data) > days else data

def calculate_liquidation_risk(utilization_rate: float) -> str:
    """根据利用率评估清算风险等级"""
    if utilization_rate > 0.95:
        return "🔴 高风险 — 接近清算线，建议立即关注"
    elif utilization_rate > 0.80:
        return "🟡 中等风险 — 利用率偏高，持续监控"
    else:
        return "🟢 低风险 — 当前安全"

# Tool Schema 定义（供 Claude Code Tool Use 注册）
TOOLS_SCHEMA = [
    {
        "name": "get_protocol_tvl",
        "description": "获取指定 DeFi 协议的当前总锁仓价值(TVL)及多链分布",
        "parameters": {
            "type": "object",
            "properties": {
                "name": {"type": "string", "description": "协议名称，如 'aave-v3', 'compound-v3'"}
            },
            "required": ["name"]
        }
    },
    {
        "name": "get_lending_data",
        "description": "获取借贷协议的详细数据：利用率、借贷总额、清算数据",
        "parameters": {
            "type": "object",
            "properties": {
                "name": {"type": "string", "description": "DefiLlama 借贷协议标识，如 'aave-v3', 'compound-v3'"}
            },
            "required": ["name"]
        }
    },
    {
        "name": "calculate_liquidation_risk",
        "description": "根据给定利用率数值评估清算风险等级（低/中/高）",
        "parameters": {
            "type": "object",
            "properties": {
                "utilization_rate": {"type": "number", "description": "利用率，如 0.72 表示 72%"}
            },
            "required": ["utilization_rate"]
        }
    }
]
```

> **关键设计**：工具函数与 Tool Schema 分离，Schema 供 Claude Code 注入 System Prompt 实现自动 Function Call 调度；工具函数内部包含超时和异常处理，确保 API 不可用时返回降级信息而非崩溃。

### 3.5 容错与降级设计

Agent 系统依赖外部 API（DefiLlama）和 LLM 推理服务（DeepSeek V4），任何一个环节不可用都会影响用户体验。因此设计了多层容错机制：

```
正常路径： 用户 → DeepSeek V4 → Tool Call → DefiLlama API ✓ → 返回分析结果
           ⋮                    ⋮            ⋮
降级路径1：                      ⋮            ✗ API 超时/500 → 返回缓存数据 + "数据可能滞后"标识
降级路径2：                      ✗ LLM 不可用 → 前端直接展示缓存仪表盘数据（无 NL 分析）
降级路径3：                      ✗ 全部不可用 → 显示维护中提示 + 最后更新时间
```

| 故障点 | 检测方式 | 降级行为 | 恢复策略 |
|---|---|---|---|
| DefiLlama API 超时（>10s） | `httpx.TimeoutException` | 返回最近一次缓存数据，标注"非实时" | 指数退避重试（1s → 2s → 4s），最多 3 次 |
| DefiLlama API 5xx | HTTP status ≥ 500 | 同超时降级 | 间隔 30s 后重试 |
| DeepSeek V4 不可用 | API 返回 5xx / 超时 | 关闭 NL 问答，前端切换为纯仪表盘模式 | 心跳检测（每 60s），恢复后自动开启 NL |
| 用户输入含无关/恶意内容 | 关键词过滤 + LLM 判断 | 返回引导提示："请描述您想了解的 DeFi 协议信息" | — |

### 3.6 缓存策略

DefiLlama API 存在频率限制（部分接口 1次/10秒），且多用户并发查询时需要避免重复请求，因此引入两层缓存：

| 缓存层 | 存储 | TTL | 内容 |
|---|---|---|---|
| L1 — 内存缓存 | Python `cachetools.TTLCache` | 60s | 热门协议（Top 10 TVL）的基础数据 |
| L2 — 数据库缓存 | SQLite (本地) / 微信云开发数据库 | 300s | 所有查询过的协议数据 + 历史走势摘要 |

```
请求流程：
用户查 "Aave" → 检查 L1 内存缓存 → 命中？直接返回
                                    → 未命中 → 检查 L2 数据库 → 命中？返回 + 回填 L1
                                                                   → 未命中 → 调 DefiLlama API → 写入 L1 + L2 → 返回
```

## 五、心得体会

### 5.1 Vibe Coding 对开发效率的提升

在本项目的开发过程中，**Claude Code + DeepSeek V4** 的 Vibe Coding 范式展现了显著优势：

1. **原型验证速度极快**：用自然语言描述"我需要一个 Agent 能查 DeFi 协议的健康度"，Claude Code 即可自动生成数据接入代码、Tool Schema 定义和 Agent 编排逻辑。传统手动开发需要 3-5 天的工作量，在 Vibe Coding 模式下 3-5 天内即可完成更完善的 MVP（含容错和缓存）。

2. **需求迭代成本大幅降低**：新增功能（如添加预警推送、补充多链对比）只需用自然语言描述变更，Claude Code 在现有代码上下文中自动调整工具链和后端逻辑，无需手动修改整个代码架构。

3. **降低跨领域开发门槛**：开发者无需同时精通区块链 RPC、DefiLlama API 细节、小程序开发和 Prompt Engineering——Claude Code 在跨领域知识桥接中起关键作用，将开发者的精力集中于产品设计和用户体验。

### 5.2 智能体的本质是"赋能而非替代"

Agent 的角色不是替代金融分析师，而是一个**超低成本的风险监控助手**：
- 它负责数据采集、初步分析、异常预警
- 分析师/用户的工作从"到处找数据"转变为"基于 Agent 的输出做深度判断和决策"
- 这正符合普惠金融的核心理念——让更多人获得信息，而不是让少数人垄断信息

### 5.3 DeFi + AI Agent = 天然的融合

链上数据的三个特性使其天然适合 AI 代理处理：
- **公开透明**：数据可自由获取，无需权限
- **结构化**：链上事件、合约状态以标准化格式存在
- **实时性**：区块产生即产生数据流

这意味着 DeFi 领域可能是 AI Agent 最成熟的落地场景之一。

### 5.4 局限与反思

1. **数据延迟**：链上数据依赖区块确认时间（~12s/block）和 DefiLlama API 刷新频率（通常 30s-5min 不等），无法做到真正的实时响应。已在文档中明确标注而非承诺"实时"。
2. **推理可靠性**：DeepSeek V4 在金融语义理解上表现优秀，但在需要精确数值计算的场景（如清算阈值比对）仍必须结合传统代码逻辑——LLM 做"理解和表达"，代码做"计算和校验"，二者不可混淆。
3. **部署限制**：微信小程序审核机制对金融类应用敏感，体验版可绕过但无法正式发布。已设计 Web H5 作为 Plan B，同一套 API 后端支持多前端切换。
4. **数据源依赖**：DefiLlama API 是第三方免费服务，存在服务变更或限流的风险。已通过 L1/L2 双层缓存 + 降级策略降低影响，但长期仍需考虑接入备选数据源（如 The Graph 子图查询）。
5. **Vibe Coding 的现实边界**：Claude Code + DeepSeek V4 能显著加速原型开发，但**不可替代工程判断**——缓存策略选择、容错链路设计、小程序审核策略等决策仍需人工主导。Vibe Coding 的角色是"效率放大器"而非"全自动工厂"。
